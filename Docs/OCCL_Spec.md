# OCCL Specification v0.2

## Open Connector & Component Library

**Version:** 0.2
**Status:** Draft
**Last Updated:** July 2026

---

# 1. Introduction

The **Open Connector & Component Library (OCCL)** is an open-source repository of automotive (and adjacent) electrical components, connectors, and supporting assets.

OCCL serves two primary purposes:

* A **human-readable reference library**, similar to a wiki.
* A **machine-readable library** for software tools such as wiring harness builders, pinout viewers, documentation generators, and validation software.

The guiding principle of OCCL is that it describes **manufacturer-defined hardware**, while user-created harnesses describe **how that hardware is used**.

---

# 2. Design Philosophy

OCCL intentionally separates immutable manufacturer data from project-specific wiring information.

A component definition should answer:

> **"What did the manufacturer build?"**

A harness project should answer:

> **"How has the user wired this component into their system?"**

This separation allows a single OCCL component definition to be reused across countless independent projects.

---

# 3. Repository Structure

The current structure is intentionally flat — one file per part, named by part number, grouped by manufacturer:

```
OCCL/
│
├── Components/
│   ├── AEM/
│   │   ├── 30-8407.json
│   │   └── 30-8408.json
│   │
│   ├── Bosch/
│   │   └── PCE-XL.json
│   │
│   ├── CMS/
│   │   ├── CM200xx.json
│   │   └── iM-225.json
│   │
│   └── HPT/
│       └── COR-EV-000.json
│
├── Connectors/
│   └── TE/
│       ├── 6437288-1.svg
│       ├── DTM06-12SA.svg
│       ├── DTM06-12SA.json      (optional metadata, see §6)
│       └── ...
│
├── scripts/
│   ├── build_occl_index.py       (generates OCCL-index.json, see §4)
│   └── README.md
│
├── .github/workflows/
│   └── build-index.yml           (runs the script on every push)
│
├── OCCL-index.json               (generated — see §4)
│
├── Docs/
│   └── OCCL_Spec.md
│
└── README.md
```

A component's file is always `Components/<Manufacturer>/<PartNumber>.json`. A connector's assets are always `Connectors/<Manufacturer>/<PartNumber>.svg`, with an optional sibling `<PartNumber>.json` for enrichment (see §6).

> This is a change from earlier drafts of this spec, which proposed nested per-part folders (e.g. `Connectors/TE/DTM15-12PA/connector.json`). That structure was never adopted; the flat layout above is what the repository and all consuming tools (e.g. POSH) actually implement.

# 4. Machine-Readable Index (OCCL-index.json)

Consumers of OCCL (harness builders, pinout viewers, etc.) should **not** query the GitHub API or crawl the repository directly — unauthenticated API access is rate-limited, and directory listings aren't a stable contract.

Instead, `OCCL-index.json` at the repository root is the single, authoritative manifest. It's generated by `scripts/build_occl_index.py`, which runs automatically via a GitHub Action on every push to `Components/**` or `Connectors/**`, and is committed back to `main`.

```jsonc
{
  "format": "OCCL-Index",
  "version": 1,
  "components": [
    { "manufacturer": "HPT", "partNumber": "COR-EV-000", "path": "Components/HPT/COR-EV-000.json" }
  ],
  "connectors": [
    { "manufacturer": "TE", "partNumber": "DTM06-12SA", "svg": "Connectors/TE/DTM06-12SA.svg", "json": "Connectors/TE/DTM06-12SA.json" },
    { "manufacturer": "TE", "partNumber": "6437288-1", "svg": "Connectors/TE/6437288-1.svg", "json": null }
  ],
  "invalid": [
    { "path": "Components/Bosch/PCE-XL.json", "reason": "invalid JSON: Expecting value: line 1 column 1 (char 0)" }
  ]
}
```

## Validation

Because a component's `.json` is the *only* source of its pin data, entries are validated before being listed under `components`. A component is included only if it:

* Parses as valid JSON.
* Has `manufacturer`, `partName`, `partNumber`, and a non-empty `connectors` array.
* Each connector has `manufacturer`, `partNumber`, and a non-empty `pins` array.
* Each pin's `pin` field is a **string** (see §8).

Anything that fails is excluded from `components` and listed under `invalid` with a specific reason instead — nothing is silently dropped. This means incomplete or placeholder component files can be committed at any time without breaking the library for everyone else; they simply won't be pickable until they pass validation.

Connector `.svg`/`.json` files are treated more leniently — see §6.

# 5. Components

A component represents a manufactured device.

Examples include:

* BMS
* VCU
* PDU
* Inverter
* Charger
* Display
* Sensor

Component definitions contain information that is considered effectively immutable.

Typical fields:

* `manufacturer`
* `partName`
* `partNumber`
* `categories`
* `connectors` — an array, each with `label`, `manufacturer`, `partNumber`, and `pins`

Component definitions intentionally **do not** contain vehicle-specific wiring information.

---

## Example

```json
{
    "manufacturer": "HPT",
    "partName": "CORE EV VCU",
    "partNumber": "COR-EV-000",
    "categories": ["VCU"],

    "connectors": [
        {
            "label": "A",
            "manufacturer": "TE",
            "partNumber": "6437288-1",
            "pins": [
                {
                    "pin": "1",
                    "function": "ECU GROUND / BATTERY NEGATIVE",
                    "notes": "ECU GROUND"
                }
            ]
        }
    ]
}
```

`label` distinguishes multiple connectors on the same component (e.g. `"A"`, `"B"`, `"C"`) and is required whenever a component has more than one connector.

---

# 6. Connectors

Connectors are referenced by manufacturer/part number from one or more components.

A connector is defined by two files, both optional in different ways:

* `Connectors/<Manufacturer>/<PartNumber>.svg` — the connector graphic. If missing, consuming tools should render a placeholder rather than fail (POSH does this today).
* `Connectors/<Manufacturer>/<PartNumber>.json` — optional metadata (position count, gender, mating info, etc.). If missing or malformed, it's simply treated as absent; it never blocks a component from being usable.

Connector definitions intentionally **do not** define component-specific pin *functions* — those live on the component, since the same physical connector can be wired to different functions on different devices. A connector's own `.json`, when present, is limited to physical/mechanical metadata about the part itself.

---

# 7. Connector Views (Long-Term Vision)

> **Current status:** OCCL currently stores exactly one SVG per connector (`<PartNumber>.svg`). The multiple-view convention below is part of the long-term vision but is not yet adopted by the repository layout, the index builder, or any consuming tool.

Different connector views communicate different information. The long-term plan is for OCCL to store each view as a discrete SVG:

```
component-face.svg
harness-front.svg
harness-rear.svg
```

These would be considered distinct engineering drawings rather than duplicates — used respectively for PCB/component-side identification, connector mating, and harness assembly/pinning. Adopting this will require a repository layout change (likely the nested per-part folder structure originally proposed in earlier drafts of this spec) and is intentionally deferred until there's real demand for more than one view per part.

# 8. Pins

Each physical cavity is represented by exactly one pin object.

```json
{
    "pin": "1",
    "function": "High Side Driver 1",
    "notes": "20 Amp Max"
}
```

**`pin` must be a string.** Some connectors use purely numeric cavity numbering (`"1"`, `"2"`, ...); others use an alphanumeric grid (`"A1"`, `"B15"`, `"M4"`, ...). Storing `pin` as a string regardless of which convention applies keeps the field consistently typed across the whole library, and allows tools to reliably parse alphanumeric grid patterns (see §8.1) without needing to special-case numeric-vs-string pin IDs. This is enforced by the index validator (§4); a component with numeric `pin` values will be excluded and flagged.

Even when multiple cavities are electrically common, each cavity is represented independently — one physical cavity, one pin object. This preserves the physical topology of the connector while allowing renderers to merge identical rows for presentation if desired.

## 8.1 Alphanumeric Grid Pins

Connectors organized into physical banks (e.g. a 2×15 grid, or a 4×12 grid) are expected to use pin IDs that combine a letter group with a numeric row (or vice versa) — e.g. `"A1"`...`"A15"`, `"B1"`...`"B15"`, or `"A1"`...`"M1"`, `"A2"`...`"M2"`, etc.

Consuming tools (e.g. POSH) can detect this pattern purely from the pin IDs already present — no additional schema or metadata is required — and render banked connectors as multiple side-by-side Pin/Function column groups instead of one long list, provided the naming forms a complete rectangle (every group has an entry for every row, with no gaps or typos). A single stray or missing pin ID breaks this detection for the whole connector, so consistency in pin naming within a bank matters.

# 9. Function vs Signal

OCCL uses the field name:

```
function
```

rather than:

```
signal
```

Reasoning:

Not every electrical connection carries a signal.

Examples include:

* Power
* Ground
* Shield
* Wake line
* High-side driver
* Configuration strap

The term **function** more accurately describes the purpose of the pin.

---

# 10. Notes

Manufacturer comments are stored in:

```
notes
```

Examples:

* 20 Amp Max
* Ground for Unit IDs 3–8
* Reserved
* Leave Open
* Unterminated

These are informational only.

# 11. Harness Projects

Harness projects exist outside OCCL.

They reference OCCL components but never modify them. The exact schema is owned by whichever tool creates the harness project (e.g. POSH's `.posh` format) rather than by OCCL — the illustrative example below is conceptual, not a literal schema OCCL prescribes:

```json
{
    "component": "AEM/BMS Mini Master",
    "instance": "BMS1",

    "connections": [
        {
            "connector": "J1",
            "pin": "1",

            "colour": "red",
            "gauge": "20 AWG",

            "endpoint": "Fuse F1",
            "notes": "Battery supply"
        }
    ]
}
```

# 13. Mating Connectors (Long-Term Vision)

> **Current status:** not yet implemented. Component definitions today list only the connector(s) physically mounted on the component (via `connectors[]`). There is currently no field distinguishing a component-side connector from its recommended mating harness connector.

The long-term plan is for component definitions to include both:

* The connector physically mounted on the component.
* The recommended mating harness connector.

```json
{
    "componentConnector": "TE/DTM15-12PA",
    "harnessConnector": "TE/DTM06-12SA"
}
```

This would make each component page self-contained as a reference while still allowing connector definitions to remain independent reusable objects. Future versions may also support multiple compatible mating connectors.

# 15. Guiding Principles

* One physical cavity equals one pin object.
* Pin IDs are always strings.
* Components describe manufacturer intent.
* Harnesses describe user implementation.
* Connectors are reusable library objects.
* A connector's SVG and JSON are both optional independently; a missing SVG should degrade to a placeholder, not an error.
* `OCCL-index.json` is the single contract between the repository and any consuming tool — nothing should query the GitHub API or crawl directories directly.
* Invalid or incomplete contributions are excluded and flagged, never silently dropped.
* OCCL prioritizes clarity, completeness, and contributor friendliness over aggressive normalization.
* Contributors should be able to create useful entries with only the essential information; richer metadata can be added over time.

---

# 16. Long-Term Vision

OCCL aims to become a community-maintained reference library for electrical components and connectors.

The library should be useful both to humans seeking documentation and to software consuming structured data.

By maintaining a clear separation between manufacturer-defined hardware and user-defined wiring, OCCL enables reliable reuse across projects while remaining approachable for contributors.

The long-term goal is not simply to document parts, but to provide an open foundation upon which wiring harness design, documentation, visualization, and validation tools can be built.

# Appendix A — Design Decisions

This appendix records significant architectural decisions made during the design of OCCL.

The purpose is to preserve the reasoning behind the library's structure so that future contributors understand not only *what* decisions were made, but *why*.

---

## DD-001 — Components are immutable

Component definitions describe the hardware as manufactured.

They intentionally do not contain vehicle- or project-specific information such as wire colours, wire gauges, routing, endpoints, or user notes.

### Rationale

A single component should be reusable across many independent projects without modification.

Vehicle-specific information belongs in a harness project rather than the component library.

---

## DD-002 — Components reference both connector halves

**Status: not yet implemented.** See §13. Current component definitions list only the connector(s) mounted on the component itself.

Each component *should eventually* reference:

* the connector fitted to the component
* the recommended mating harness connector

### Rationale

Although the mating connector is technically a separate part, users may consult OCCL as a reference library to determine which connector mates with a particular component.

Including both part numbers makes component pages self-contained while preserving connector definitions as reusable library objects.

---

## DD-003 — Connectors are first-class library objects

Connectors exist independently of components.

A single connector may be referenced by many different devices.

### Rationale

This avoids duplication of connector graphics and metadata while allowing connectors to evolve independently of component definitions.

---

## DD-004 — SVG views are discrete assets

**Status: not yet implemented.** See §7. Currently exactly one SVG is stored per connector.

The long-term plan is for different viewpoints of a connector to be stored as separate SVGs, e.g.:

* Component face
* Harness front
* Harness rear

### Rationale

These drawings communicate different information and are considered distinct engineering views rather than alternate renderings of the same asset.

Although this increases authoring effort, it produces a more complete and useful reference library.

---

## DD-005 — One physical cavity equals one pin object

Each physical cavity is represented individually within component definitions.

Even if multiple cavities are electrically common, each receives its own pin object.

### Rationale

The library models the physical connector rather than the schematic representation.

This enables accurate highlighting, validation, and harness generation while allowing renderers to merge identical rows when presenting pin tables.

---

## DD-006 — Pin "function" is preferred over "signal"

Pin definitions use the field name:

```
function
```

rather than:

```
signal
```

### Rationale

Not all pins carry signals.

Examples include:

* Power
* Ground
* Shield
* High-side outputs
* Configuration straps

The term *function* more accurately represents the purpose of a connector cavity.

---

## DD-007 — OCCL is a reference library, not an ECAD database

The scope of OCCL is intentionally limited.

The library focuses on documenting:

* Components
* Connectors
* Pin functions
* Connector graphics

Detailed connector ecosystem data such as:

* Terminals
* Seals
* Wedges
* Crimp tooling
* Wire insulation compatibility

is considered optional future metadata.

### Rationale

Keeping the required metadata lightweight lowers the barrier to contribution while allowing the library to grow organically.

---

## DD-008 — Components describe hardware, harnesses describe implementation

OCCL stores manufacturer intent.

Harness projects store implementation details.

### Rationale

Separating these responsibilities prevents duplication while allowing a single component definition to be reused across many independent harness projects.

---

## DD-009 — Connector graphics are independent assets

SVG files are not embedded within component definitions.

Components reference connector part numbers rather than owning connector graphics.

### Rationale

Many different components share identical connectors.

Keeping connector graphics independent minimizes maintenance and ensures consistency across the library.

---

## DD-010 — Human readability is a primary design goal

OCCL is intended to function as both:

* a machine-readable component library
* a human-readable technical reference

Where appropriate, component definitions may include convenience information (such as recommended mating connector part numbers) even when that information can be inferred elsewhere.

### Rationale

A contributor or user browsing the library should be able to answer common engineering questions from a single component page without excessive cross-referencing.

---

## DD-011 — Contributor effort should scale with available information

The minimum useful OCCL contribution should require only:

* Component identification
* Connector identification
* Pin functions
* Connector graphics

Additional metadata should be additive rather than mandatory.

### Rationale

Lowering the barrier to entry encourages contributions while allowing future contributors to enrich existing entries over time.

---

## DD-012 — Pin IDs are always strings

Pin identifiers (the `pin` field) are stored as strings, never numbers, even for connectors using purely numeric cavity numbering.

### Rationale

Some connectors use alphanumeric grid numbering (e.g. `"A1"`, `"M4"`) to represent physical banks of pins. Consuming tools can detect and render this structure automatically purely from the string pattern of existing pin IDs, without any additional schema. Requiring `pin` to always be a string — rather than allowing numeric IDs for simple connectors and strings only for banked ones — keeps the field uniformly typed across the entire library and removes the need for consuming code to special-case one representation or the other.

---

## DD-013 — OCCL-index.json is the sole machine contract

Consuming tools read `OCCL-index.json`, generated automatically from repository contents, rather than querying the GitHub API or crawling directories directly.

### Rationale

Unauthenticated GitHub API access is rate-limited (60 requests/hour), which makes runtime directory queries impractical for any tool loading the library on every use. A single generated manifest, rebuilt automatically on every relevant push, gives consumers one fast, stable fetch regardless of library size, and gives the library a natural place to enforce validation (§4) without burdening every consumer with re-implementing it.
